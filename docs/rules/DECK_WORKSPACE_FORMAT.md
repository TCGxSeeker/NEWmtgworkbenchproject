# Deck Workspace Format

## Purpose

`.mtgwdeck.json` is the native saved workspace format for MTG Workbench deckbuilding. It stores editable deck state for future UI and command workflows.

Import/export formats such as plain text, CSV, Archidekt-style exports, or future external decklist formats are boundary formats. They should be converted into or out of the native workspace rather than treated as the saved source of truth.

## Required Workspace Fields

- `schema_version`: integer format version, currently `1`.
- `deck_id`: stable workspace identifier.
- `name`: deck name.
- `format`: deck format, such as `commander`.
- `tags`: workspace-level labels.
- `commanders`: commander-zone entries.
- `mainboard`: main deck entries.
- `maybeboard`: cards held outside the active main deck.
- `saved_state`: placeholder for dirty/saved-state tracking.

## Optional Workspace Fields

- `created_at`
- `updated_at`
- `notes`
- `metadata`

`metadata` may record import origin, source path, or other non-card workspace context.

## Deck Entry Fields

Each entry stores:

- `entry_id`
- `quantity`
- `input_name`
- `display_name`
- `oracle_id`
- `selected_printing_id`
- `zone`
- `categories`
- `tags`
- `imported_category`
- `normalized_category`
- `generic_category_hint`
- `deck_specific_primary_role`
- `secondary_tags`
- `category_origin`
- `notes`
- `pinned`
- `foil`
- `date_added`
- `is_unresolved`

`zone` must be one of `commander`, `mainboard`, or `maybeboard`.

## Deck Entry Category Metadata

`categories` remains the current grouping field. The category metadata fields preserve provenance and future analysis context:

- `imported_category`: original category/header from an import or user action.
- `normalized_category`: canonical category when the local taxonomy matched a canonical name or alias.
- `generic_category_hint`: broad non-authoritative category hint for grouping or later review.
- `deck_specific_primary_role`: future placeholder for the role that counts in this exact deck context.
- `secondary_tags`: non-counting supporting labels.
- `category_origin`: category source, such as `imported`, `user`, `normalized`, `inferred`, `taxonomy_default`, or `unknown`.

V0 does not enforce primary roles, auto-categorize cards, or use these fields for role counting.

## Unknown Card Preservation

Unknown cards are preserved by keeping `input_name`, setting `display_name`, `oracle_id`, and `selected_printing_id` to `null` when needed, and marking `is_unresolved` as `true`.

The loader must report malformed workspace files clearly, but it must not guess unknown card identities.

## Mutation Behavior

Deck Workspace Mutations v0 edits `DeckWorkspace` objects in memory and returns the same workspace for caller chaining.

Supported v0 mutations:

- Add an entry to commander, mainboard, or maybeboard.
- Remove an entry by `entry_id`.
- Increase or decrease quantity.
- Remove an entry when quantity decreases to zero.
- Move an entry between zones.
- Set an entry as commander.
- Replace categories.
- Set or clear category metadata.
- Add, remove, replace, or clear secondary category tags.
- Replace, add, or remove tags.
- Set or clear notes.

Missing `entry_id` values are generated by the mutation helper. Caller-supplied `entry_id` values are preserved.

Category metadata edits are explicit. They preserve the existing `categories` grouping field unless the caller uses the separate category replacement helper.

Normalized category values may be validated against a supplied local category taxonomy. Without a taxonomy, mutation helpers preserve caller-supplied values without guessing or auto-normalizing.

## Duplicate Add Policy

V0 merges duplicate adds only when the new card clearly matches an existing logical entry:

- Same zone.
- Same `oracle_id` when present.
- Same `selected_printing_id`.
- Same display/input-name fallback when no `oracle_id` exists.
- Same categories.
- Same category metadata.
- Same foil flag.
- Same unresolved status.

If category, printing, foil status, zone, or identity is materially different, the add creates a separate entry. Unresolved entries only merge with clearly identical unresolved entries.

## Dirty State And Timestamps

Every mutation updates `updated_at` and sets `saved_state.is_dirty` to `true` when `saved_state` is present.

Loading a workspace through `load_workspace` returns a clean workspace. Successful `save_workspace` marks `saved_state.is_dirty` as `false` before writing.

Low-level string serialization does not represent a full workspace session layer; it is a JSON conversion helper.

## Plain Text Import/Export V0

Plain text is an import/export boundary format, not native workspace state.

Supported import line formats:

- `1x Card Name`
- `1 Card Name`
- `Card Name`

Supported section headers:

- `Commander`
- `Commanders`
- `Mainboard`
- `Main Deck`
- `Deck`
- `Maybeboard`
- `Maybe Board`
- `Sideboard`

`Sideboard` is treated as `maybeboard` in v0 because the native model does not have a sideboard zone.

Conservative category headers may be applied when they exactly match known deckbuilder categories such as `Ramp`, `Draw`, `Removal`, `Interaction`, `Protection`, `Lands`, `Creatures`, `Instants`, or `Sorceries`.

When a local category taxonomy is supplied to the importer, canonical category names and aliases can populate `normalized_category` while preserving the original `imported_category`. Unknown labels are not guessed.

Export uses `Commander`, `Mainboard`, and `Maybeboard` sections with lines formatted as `1x Card Name`. Export prefers `display_name` and falls back to `input_name` for unresolved cards.

## Deferred In V0

The mutation and import/export layers do not perform full Commander legality validation, report generation, recommendation logic, UI behavior, pricing checks, live API calls, telemetry, or advanced external deckbuilder format support.

## Versioning

Schema changes should increment `schema_version` and keep migration behavior explicit. Version `1` is the first deckbuilder foundation workspace format.
