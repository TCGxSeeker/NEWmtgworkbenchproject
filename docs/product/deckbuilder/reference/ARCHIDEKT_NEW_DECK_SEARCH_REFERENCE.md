# Archidekt New Deck And Search Reference

## Purpose

Capture user-provided Archidekt screenshots as product reference for smooth
deck creation and search/add pathing. Use these notes for capability and
interaction planning only; do not clone styling, branding, layout density,
advertising, account features, or service-specific behavior.

## New Deck Path Cues

- A visible `New deck` action should be easy to reach from the starting surface.
- Initial creation should prioritize the few required fields:
  - deck name
  - format
  - optional Commander bracket or power target
- Optional setup belongs behind progressive disclosure, such as collapsed
  sections for card packages, folder/storage options, and category options.
- Local-only MTG Workbench should not copy hosted visibility controls such as
  private, unlisted, public profile, or theorycrafted/social listing states.
- The primary create action should be visually distinct and placed where the
  form naturally ends.

## Empty Deck Workspace Cues

- An empty deck should still open into the real deckbuilder workspace.
- The empty state should point users toward import, quick add, and card search
  without becoming a large instructional dashboard.
- Header facts should stay compact: format, size, estimated cost if available,
  target bracket/power if set, tags, and saved state.
- Import and more-menu actions are useful near the deck header but should not
  crowd the main card workspace.

## Search Workspace Cues

- Search/add can appear as a focused overlay or expanded work surface while the
  deck remains dimmed or contextually visible behind it.
- Core tabs should separate workflows rather than mixing them:
  - card search
  - syntax search
  - future landbase helper
  - future combo helper
  - future manually supported recommendation helper
- Advanced options and filter/sort controls should remain collapsed by default.
- A full-width search box with a clear search action works well for deliberate
  card lookup.
- Search result counts are useful when shown quietly.
- Clear/reset should be available without competing with the primary search
  action.

## Search Result Cues

- Card-image grid results are valuable for recognition once local image support
  exists.
- Each result should expose quick add/increase, decrease, and more/details
  actions without requiring navigation away from the search surface.
- Added cards should show immediate visible feedback, such as:
  - highlighted border
  - `Copies: N` badge
  - quantity marker
  - plus/minus controls
- Price rows are not required for v0 and should remain absent unless backed by
  a verified local snapshot and a separate pricing surface decision.

## MTG Workbench Translation

Use these references to guide:

- New Deck flow planning.
- Empty workspace onboarding.
- Search/add overlay structure.
- Search mode tabs.
- Collapsed advanced controls.
- Card-image result grid design.
- In-result quantity feedback.

Keep MTG Workbench local-first, deck-centered, and calmer than the reference
screens. The deck view remains the product center; search is a supporting
workspace.

## Do Not Carry Forward

- Do not copy private/unlisted/theorycrafted hosted visibility controls.
- Do not add ads, hosted user profiles, social listing states, or account
  features from the reference.
- Do not add EDHREC, price-source, live import, or other online integrations
  from these screenshots.
- Do not surface recommendation helpers before their own approved slice.
- Do not make card search the main product surface.
